## 名词解释（新手先读这里）

在阅读本文之前，先了解几个会反复出现的概念：

| 名词 | 一句话解释 | 类比 |
|------|----------|------|
| **localStorage** | 浏览器里的小数据库，页面关闭后数据不丢 | 就像浏览器给你的一个"抽屉"，只能本机访问 |
| **CDN** | 内容分发网络，在全球各地有缓存服务器 | 就像各地有仓库，取货很快，但更新有延迟 |
| **CORS** | 跨域请求安全机制，限制网页访问其他域名 | 就像海关，遇到"可疑"请求就要先盘问（预检） |
| **API** | 程序之间通信的接口 | 就像后台的服务窗口，你递请求，它给数据 |
| **Token** | 一串密码字符，用来证明身份 | 就像工牌，刷了才能进系统操作 |
| **SHA / SHA-256** | 一种算法，把任意文字变成固定长度的"指纹" | 就像指纹，同样的输入永远产生同样的输出，但不能从指纹反推输入 |
| **Base64** | 把二进制数据转成纯文本的编码方式 | 就像把图片打包成快递盒，方便运输 |
| **JSON** | 一种轻量级的数据交换格式 | 就像一张格式统一的卡片：`{"名字": "张三", "年龄": 25}` |
| **异步 (async/await)** | 代码不等结果，继续干别的事 | 就像点了外卖，不站门口等，先做自己的事，到了再取 |
| **回调 (Promise/.then)** | 异步任务完成后要执行的代码 | 就像"外卖到了通知我"，不用一直盯着 |

### 关键理解

```

管理员浏览器 ──API读取──▶ GitHub仓库 ← 即时但需 Token 认证
```

## 二、技术栈

| 层级 | 技术 | 为什么选它 |
|------|------|----------|
| **托管** | GitHub Pages | 免费、自动 HTTPS、和仓库绑定 |
| **前端** | 原生 HTML + CSS + JavaScript | 零依赖，一个文件搞定，不用装 Node.js |
| **存储** | 浏览器 localStorage | 每个浏览器独立，关掉页面后数据还在 |
| **云端** | GitHub REST API v3 | 个人 Token 免费，数据有 git 版本历史 |
| **认证** | PBKDF2-SHA-256 加盐派生 + GitHub PAT Token | 密码防误操作，Token 才是云端写入权限边界 |
| **公共读** | raw.githubusercontent.com CDN | 访客不用登录就能看数据 |

### 为什么不用数据库而是存 GitHub 文件？

- 不需要买服务器、不需要配数据库
- GitHub 免费提供文件存储 + 版本历史
- 修改 `data.json` 就像修改其他文件一样简单
- 所有修改都有 commit 记录，可以回滚

## 三、GitHub 配置详解

### 3.1 页面部署 (GitHub Pages)

简单说：把 `index.html` 推到一个 Public 仓库的 `main` 分支，GitHub 就自动帮你把网页挂到互联网上。

```bash
git remote add origin https://github.com/<用户名>/<仓库名>.git
git branch -M main
git push -u origin main
```

然后去仓库 Settings → Pages → Source: main → Save，等约 1 分钟。

**访问地址**：`https://<用户名>.github.io/<仓库名>/`

### 3.2 API 令牌 (Fine-grained PAT)

有了令牌才能通过代码读写 GitHub 上的文件。Fine-grained 比 Classic 安全，因为可以限制到**单个仓库**的**单个权限**。

**创建步骤**：
1. 打开 https://github.com/settings/tokens?type=beta
2. Generate new token → Fine-grained token
3. Token name: `命令管理器`
4. Repository access: **Only select repositories** → 选中你的仓库
5. Permissions → Contents → **Read and write**（这已经是全部需要的权限了）
6. 生成后**立刻复制**，页面一关就看不到了

**在代码里怎么用**：

```javascript
// 令牌存在浏览器里
localStorage.setItem('github_token', 'github_pat_xxxxxx');

// 每次调 GitHub API 时放在请求头里，相当于递工牌
fetch(url, {
    headers: {
        Authorization: 'Bearer github_pat_xxxxxx',
        Accept: 'application/vnd.github.v3+json'
    }
});
```

### 3.3 API 文件读写操作

项目中只用到 GitHub API 的两个操作：

#### A. 读取文件 (GET)

```
GET https://api.github.com/repos/用户名/仓库名/contents/data.json

返回内容：
{
    "content": "base64编码的文件内容",   ← 需要解码才能用
    "sha": "abc123def456...",           ← 文件当前的"指纹"，写文件时必须传
    "size": 12345,
    ...
}
```

#### B. 写入文件 (PUT)

```
PUT https://api.github.com/repos/用户名/仓库名/contents/data.json

请求体：
{
    "message": "同步命令数据",
    "content": "base64编码后的新文件内容",
    "sha": "abc123def456...",   ← 必须传当前文件的 SHA
    "branch": "main"
}
```

#### C. SHA 是什么？为什么需要它？

SHA 就像是给文件拍了一张"指纹快照"。GitHub 用这个机制保证**不会有两个人的修改互相覆盖**：

1. 你读取文件 → 记下 SHA
2. 你准备写文件 → 带上 SHA
3. GitHub 检查：你带的 SHA 和当前文件的 SHA 一致吗？
   - 一致 → 说明文件没被别人改过，允许写入 ✅
   - 不一致 → 说明你读走后别人改过了，拒绝写入，返回 **409 Conflict** ❌

这叫做**乐观锁**（Optimistic Locking），就是"先假设不会有冲突，出冲突再说"。

代码会把上次成功同步的 SHA 记录为共同版本。若推送前发现远端 SHA 已变化，会停止自动覆盖并显示冲突；只有用户在手动同步时明确确认，才允许用本地数据覆盖云端。

#### D. Base64 编码（为什么需要特殊处理）

GitHub API 要求文件内容必须是 Base64 编码。但 JavaScript 原生的 `btoa()` 不支持中文，所以我们用了兼容写法：

```javascript
// 写入：把中文也正确转成 Base64
function utf8ToBase64(str) {
    return btoa(unescape(encodeURIComponent(str)));
}

// 读取：把 Base64 正确转回中文
function base64ToUtf8(str) {
    return decodeURIComponent(escape(atob(str)));
}
```

### 3.4 CDN 读取 vs API 读取（速度差异的真正原因）

这是最容易被误解的地方。两条通道**本质不同**：

`api.github.com` 走的是什么：
```
浏览器 → URL → GitHub 数据库 → 返回最新的文件内容
         ↑ ① 跨域了，先发 OPTIONS 预检请求
         ↑ ② 预检通过，再发真正的 GET 请求
         ↑ ③ 请求头里带 Authorization Token
         ↑ ④ GitHub 验证 Token 后返回数据
```
- 一次读取需要**两次 HTTP 请求**（OPTIONS + GET）
- 每次都要验证 Token
- 但返回的是**实时数据**，不存在缓存延迟

`raw.githubusercontent.com` 走的是什么：
```
浏览器 → URL → 离你最近的 CDN 节点 → 返回缓存的文件内容
         ↑ 只发一次请求，不需要 Token
```
- 只需要**一次 HTTP 请求**
- 不需要认证，没有跨域预检
- 全球 CDN 加速，延迟极低
- 但是 CDN 缓存有**延迟**，GitHub 更新后最多等 1-5 分钟

#### 为什么加了 `?_=时间戳` 还是有延迟？

时间戳 (`?_=Date.now()`) 只是告诉**你自己浏览器**"不要用本地缓存"，它管不了 CDN 服务器上的缓存。CDN 的缓存更新周期由 GitHub 控制，约 1-5 分钟。

| 特性 | API (`api.github.com`) | CDN (`raw.githubusercontent.com`) |
|------|----------------------|--------------------------------|
| 需要 Token | ✅ 需要 | ❌ 不需要 |
| HTTP 请求数 | 2 次（OPTIONS + GET） | 1 次 |
| 数据时效 | 即时 | 1-5 分钟延迟 |
| 写权限 | ✅ 有 | ❌ 只读 |
| 速度（典型） | 200-800ms | 50-200ms |
| 配额限制 | 5000 次/小时 | 无明显限制 |

**一句话总结：API 是"去仓库里取原件"，CDN 是"在家门口快递柜取复印件"。**

## 四、前端架构

### 4.1 访客/管理员双模式

打开页面立即进入**访客模式**，能看能搜能复制，但看不到管理按钮。登录后进入**管理员模式**，全部功能解锁。

**实现原理**：就是在 `<body>` 上加一个 CSS 类，然后通过 CSS 控制哪些元素可见。

```css
/* 管理员按钮：只在管理员模式下显示 */
body:not(.admin-mode) .admin-only {
    display: none !important;
}
/* 访客按钮：只在非管理员模式下显示 */
body:not(.visitor-mode) .visitor-only {
    display: none !important;
}
```

注意 `!important` —— 因为内联 style 优先级更高，需要 `!important` 确保模式切换一定能覆盖。

**页面加载时 init() 怎么分支**：

```
页面加载
  → loadCommands()        从 localStorage 加载数据（立即，不等）
  → renderCommands()      渲染界面（立即，不等）
  → 读取 admin_logged_in  判断上次是否登录了
  
     ├─ 是管理员（登录状态）
     │   → updateSyncUI()     显示同步状态
     │   → syncPull()         从 GitHub API 拉取最新数据
     │                         （数据即时准确，但需要 Token）
     │
     └─ 是访客（未登录）
         → fetchPublicData()  从 CDN 拉取最新数据
                              （快，但有延迟，不需要 Token）
```

### 4.2 登录弹窗设计

点击"🔐 登录管理"后，弹窗**始终默认显示「登录」Tab**，避免时序问题：

```
┌─────────────────────────┐
│  🔐 管理员登录           │
│  ┌──────┐ ┌──────────┐  │
│  │ 登录 │ │ 创建账号 │   │  ← 用户自己选择
│  └──────┘ └──────────┘  │
│                         │
│  用户名: [__________]    │
│  密码:   [__________]    │
│  [登录]                 │
└─────────────────────────┘
```

设计要解决的实际问题：CDN 数据可能还没加载完，不能根据"云端有没有凭据"来决定显示登录还是创建。所以给用户两个 Tab 自己选。

### 4.3 管理员凭据的跨浏览器共享

管理员账号的**密码哈希**会随命令同步一起上传到 GitHub 的 `data.json`，其他浏览器加载数据时自动读到本地：

```
A 浏览器（设好账号 + 配好 Token）：
  handleSetup() → setAdminCreds() → syncPull() + syncPush()
                                              ↓
                               data.json 写入:
                               { adminAuth, commands, ... }

B 浏览器（新打开）：
  fetchPublicData() → 读取 data.json → 发现 adminAuth
                   → syncAdminCredsFromCloud() → 存到 cloud_admin_creds
  openLoginModal() → 本地无 creds → 复制 cloud → 本地 → 显示登录表单
  handleLogin()    → 用户输入密码 → 哈希比对 → 登录成功
```

**关键点**：A 浏览器首次创建账号后，必须配置 Token 并完成一次同步（或添加/修改/删除一条命令），凭据才能上传到云端。

### 4.4 单文件结构

所有代码都在一个 `index.html` 里：

```
index.html
├── <style>   ← CSS（样式）
├── <body>    ← HTML（结构）
└── <script>  ← JS（交互逻辑）
```

**为什么这样设计？** 因为部署到 GitHub Pages 不需要任何构建工具，改了直接推，1 分钟后就能在线上看到效果。

### 4.5 数据模型

每条命令是一个对象（就是一个带标签的数据包）：

```javascript
{
    id: "550e8400-e29b-...",   // 唯一 ID，用 crypto.randomUUID() 生成
    command: "ls -la",         // 命令文本
    description: "查看所有文件", // 中文说明
    mainCategory: "Linux",     // 主分类
    subCategory: "文件操作",    // 子分类
    tags: ["文件", "查看"],     // 标签
    favorite: false,           // 是否收藏
    copyCount: 0,              // 被复制了几次
    createdAt: "2026-07-02T...", // 创建时间 (ISO 8601 格式)
    updatedAt: "2026-07-02T..."  // 更新时间
}
```

**版本兼容**：如果以后新增字段，老数据加载时会自动补默认值：
```javascript
commands.forEach(cmd => {
    if (cmd.favorite === undefined) cmd.favorite = false;
    if (cmd.copyCount === undefined) cmd.copyCount = 0;
});
```

### 4.6 localStorage 读写

localStorage 本质是浏览器提供的一个键值存储（简单理解：一个只能存字符串的字典），数据关掉浏览器也不会丢。

```javascript
// 存
localStorage.setItem('commandManagerData', JSON.stringify(commands));
// JSON.stringify 把 JS 对象变成字符串

// 取
const stored = localStorage.getItem('commandManagerData');
commands = JSON.parse(stored);  // JSON.parse 把字符串变回 JS 对象
```

**为什么要 JSON.stringify/parse？** localStorage 只能存字符串，不能直接存 JS 对象。

**重要原则**：每次修改命令后，必须同时更新 `commands` 变量和调用 `saveToStorage()`，缺一不可。

### 4.7 渲染模式

每次数据变化后，不分新旧，**全部重新画一遍**。就是：

```javascript
function renderCommands() {
    // 先把当前显示的内容清空
    grid.innerHTML = '';
    // 再把所有命令重新生成 HTML 填进去
    grid.innerHTML = commands.map(cmd => `
        <div class="command-card">...</div>
    `).join('');
}
```

- **优点**：逻辑简单，不需要判断哪些要更新哪些要删除
- **缺点**：命令多了会慢（当前 ~100 条感知不到）

### 4.8 搜索

支持**多词空格分隔**（每个词都要匹配，AND 逻辑）+ **字符级模糊匹配**：

```javascript
// 搜索 "linux 文件" → 必须同时包含 linux 和 文件
function fuzzyMatch(text, pattern) {
    // 比如搜索 "lsl" 能匹配到 "ls -la"
    // 原理：检查 pattern 的每个字符是否按顺序出现在 text 里
    let pi = 0;
    for (let i = 0; i < text.length && pi < pattern.length; i++) {
        if (text[i] === pattern[pi]) pi++;
    }
    return pi === pattern.length;
}
```

### 4.9 防抖（Debounce）

用户打字搜索时，不在每次按键后立即搜索，而是等 250 毫秒没再按键了才搜：

```javascript
let searchTimer;
function onSearchInput() {
    clearTimeout(searchTimer);         // 取消上一次的定时
    searchTimer = setTimeout(filterCommands, 250);  // 重新设定时
}
```

这样用户快速输入 "linux" 时，只会在输完最后一个 "x" 后 250ms 才触发一次搜索，而不是触发 5 次。

## 五、云同步核心逻辑

### 5.1 同步流程总览

```
页面加载：
  loadCommands()      ← ① 先渲染本地数据（快，不卡 UI）
                      
  管理员 → syncPull() ← ② 从 GitHub API 拉取，先检查共同 SHA 和本地修改状态
  访客   → fetchPublicData() ← ③ 从 CDN 拉取，按 updatedAt 判断是否更新

管理员增/删/改命令：
  saveToStorage()     ← 存本地（立即生效）
  syncPush()          ← 推云端（后台，不卡 UI）

手动同步：
  manualSync()
    → syncPull()      ← 先拉（确保 SHA 是最新的）
    → syncPush()      ← 无冲突才推送；有冲突时等待用户决定
```

### 5.2 syncPull — 从云端拉取

```javascript
async function syncPull() {
    const result = await fetchGitHubFile(token);  // 调 GitHub API 读取

    // ① 从数据中同步管理员凭据到本地
    syncAdminCredsFromCloud(result.content);

    // ② 本地有修改且远端 SHA 已变化：停止，绝不静默覆盖
    const baseSha = localStorage.getItem('lastSyncedSha');
    const dirty = localStorage.getItem('sync_dirty') === 'true';
    if (dirty && (!baseSha || baseSha !== result.sha)) return false;

    commands = result.content.commands;
    localStorage.setItem('lastSyncedSha', result.sha);
    localStorage.removeItem('sync_dirty');
}
```

**为什么比较 SHA？** 时间戳受设备时钟影响，不能可靠判断共同版本；SHA 可以准确识别远端文件是否在本机上次同步后发生过变化。

### 5.3 syncPush — 推送到云端

```javascript
async function syncPush() {
    // ① 并发控制（见 5.5）
    if (_syncing) { _pendingSync = true; return; }
    _syncing = true;

    try {
        // ② 先获取当前 SHA
        const current = await fetchGitHubFile(token);
        const sha = current?.sha;

        // ③ 远端偏离共同版本时停止，避免拿到新 SHA 后覆盖别人刚写的数据
        const baseSha = localStorage.getItem('lastSyncedSha');
        if (localStorage.getItem('sync_dirty') === 'true' && baseSha !== sha) {
            throw new Error('同步冲突');
        }
        await pushGitHubFile(commands, token, sha);

        // ④ 记录同步时间
        localStorage.setItem('lastSyncAt', Date.now());
    } catch (err) {
        showToast('同步失败: ' + err.message);
    } finally {
        _syncing = false;               // 释放锁
        if (_pendingSync) {             // 有待处理的任务？
            _pendingSync = false;
            syncPush();                 // 自动重试
        }
    }
}
```

**推送时附带管理员凭据**：

```javascript
async function pushGitHubFile(commands, token, sha) {
    const creds = getAdminCreds();
    const payload = JSON.stringify({
        commands: commands,
        updatedAt: new Date().toISOString(),
        adminAuth: creds                    // PBKDF2 算法、盐、迭代次数和派生值
    });
    // ... 上传到 GitHub
}
```

这样其他浏览器读 `data.json` 时就能自动获取管理员凭据。

### 5.4 fetchPublicData — 访客从 CDN 读取

```javascript
async function fetchPublicData() {
    const url = RAW_DATA_URL + '?_=' + Date.now();  // 时间戳防浏览器缓存
    const res = await fetch(url);  // ← 不需要 Token，一个请求搞定
    const data = await res.json();

    syncAdminCredsFromCloud(data);  // 提取管理员凭据存本地

    // 只有命令数量变化时才替换（粗糙但有效）
    if (data.commands.length !== commands.length) {
        commands = data.commands;
    }
}
```

**为什么用命令数量比对而不是时间戳？** 访客模式下没有 Token，不能用 API 读 `updatedAt` 字段。CDN 返回的数据里包含 `updatedAt`，但不同浏览器的本地 `lastSyncAt` 时间基准不同。

### 5.5 并发锁 + 排队机制 (`_syncing` + `_pendingSync`)

如果用户快速连续操作（比如连续添加两条命令），每步都触发 `syncPush()`，会导致并发问题。解决办法：

```javascript
let _syncing = false;     // 互斥锁：正在同步中
let _pendingSync = false; // 排队标记：锁释放后需要重试

async function syncPush() {
    if (!token) return;                    // 没 Token，直接放弃

    if (_syncing) {                        // 当前有同步在进行
        _pendingSync = true;               // 标记"有待处理"
        return;                            // 不阻塞 UI，返回
    }

    _syncing = true;                       // 获取锁
    try {
        // ... 真正的同步逻辑 ...
    } catch (err) {
        // ... 错误处理 ...
    } finally {
        _syncing = false;                  // 无论成不成功都释放锁
        if (_pendingSync) {                // 刚才有人排队？
            _pendingSync = false;
            syncPush();                    // 用最新数据重试
        }
    }
}
```

**工作示例**：
```
t=0  用户添加命令A → syncPush() → 获取锁，开始同步
t=1  用户添加命令B → syncPush() → _syncing=true → 设 _pendingSync=true → 返回
t=3  命令A同步完成 → finally: _syncing=false, _pendingSync=true → 自动 syncPush()
t=5  命令B同步完成
```

## 八、安全分析

> 本项目是**纯前端**单文件应用，没有后端服务器。下面解释这对安全意味着什么。

### 8.1 核心问题：登录验证在浏览器里执行

```javascript
// 这就是"整个"认证逻辑，全部在浏览器 JS 里
async function handleLogin() {
    const hash = await hashPassword(输入的密码);
    if (存本地哈希 === hash) {
        localStorage.setItem('admin_logged_in', 'true');  // ← 设个标记
        updateModeUI();  // ← 就是切个 CSS 类
    }
}
```

**这意味着什么？** 随便一个人，打开 F12 → Application → Local Storage → 把 `admin_logged_in` 改成 `"true"` → 刷新页面 → 就是管理员了。不需要知道密码。

### 8.2 F12 能看到的全部数据

| 存储 Key | 内容 | 能看到什么 |
|---------|------|----------|
| `admin_creds` | `{ username, algorithm, salt, iterations, passwordHash }` | 用户名 + PBKDF2 密码派生数据 |
| `cloud_admin_creds` | 同上 | 同上 |
| `admin_logged_in` | `"true"` | 直接篡改即绕过登录 |
| **`github_token`** | `github_pat_xxxxx` | **⚠️ 最敏感 — 持有此 Token 即可读写你的 GitHub 仓库** |
| `commandManagerData` | 全部命令 JSON | 明文可见 |
| `categoryStructure` | 分类列表 | 无关紧要 |

### 8.3 那密码还有什么意义？

**防误操作，不防攻击。** 四层防御：

```
第一层 CSS 隐藏 → 路过的人看不到管理按钮
   绕过：F12 删除 display: none 规则

第二层 JS 守卫 → 键盘快捷键也检查权限
   绕过：Console 直接调函数

第三层 密码验证 → 不知道密码不能点管理按钮
   绕过：setItem('admin_logged_in', 'true')

第四层 GitHub Token → 没有 Token 即使登录也改不了云端 ← 真正的安全边界
   绕过：窃取 Token
```

### 8.4 密码哈希细节

新账号使用随机 16 字节盐和 210,000 次 PBKDF2-SHA-256。明文密码不落盘；旧版单次 SHA-256 凭据仍可登录，并会在成功登录后自动升级。加盐和迭代只能提高离线猜测成本，不能把纯前端登录变成真正的权限边界。

### 8.5 实用建议

1. 这个工具最适合**个人使用**，不要存敏感信息
2. Token 是最高权限信物，不要分享
3. 定期更换 Token（GitHub Settings → Tokens → 撤销旧的 → 生成新的）
4. 如果 Token 泄露，直接 GitHub 上撤销就行，不会影响其他
5. 使用强密码（即使前端认证能绕过，强密码在云端凭据泄露时更安全）

## 九、已知限制

| 限制 | 实际影响 | 当下的应对 |
|------|---------|----------|
| 前端认证可绕过 | F12 设 `admin_logged_in=true` 即管理员 | 定位是防误操作，不是防攻击 |
| Token 明文在 localStorage | F12 可见 | 定期轮换，不存敏感数据 |
| 两个浏览器同时操作 | 会检测到共同 SHA 分叉 | 停止自动覆盖，由用户手动决定 |
| CDN 缓存 1-5 分钟 | 管理员推了数据，访客看不到 | 访客只是浏览用，可手动刷新 |
| 离线不能同步 | 没网时操作只存本地 | 联网后自动推送 |
| 所有代码一个文件 | 维护困难（约 1900 行） | 适合当前规模，足够用 |
| 密码派生数据公开 | 弱密码仍可能被离线猜测 | PBKDF2 加盐迭代，并使用强密码 |

## 十、常见问题排查

### Q1: A 浏览器推了数据，B 浏览器刷新看不到？
**原因**：访客模式从 CDN 读，CDN 有 1-5 分钟缓存延迟。
**解决**：等 5 分钟再刷新，或者让 B 浏览器也登录管理员模式（从 API 读，即时）。

### Q2: 登录报"未找到管理员账号"？
**原因**：
- 方案 A：A 浏览器创建账号后**没有配 Token 或没有同步过**，凭据没到云端
- 方案 B：CDN 延迟，`cloud_admin_creds` 还没加载出来
**解决**：
- A 浏览器：配好 Token → 添加/修改/删除一条命令 → 或点"☁️ 同步"按钮
- B 浏览器：等几秒再点登录

### Q3: 搜索框里自动出现 "admin" 文字？
**原因**：Chrome 的密码自动填充找到了用户名输入框。
**解决**：已修复 —— 搜索框改成了 `type="search"`，用户名输入框 id 不再包含 "username"。

### Q4: 点了同步按钮但好像没推上去？
**原因**：可能存在并发同步竞争，`_syncing` 锁阻止了第二次推送。
**解决**：已修复 —— 新增 `_pendingSync` 排队机制，并发请求会自动重试。

### Q5: data.json 有历史记录吗？
**答**：有。每次同步创建的 commit 都在你的 GitHub 仓库的 Commit History 里，可以回滚。
