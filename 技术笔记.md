## 二、技术栈

| 层级 | 技术 | 说明 |
|------|------|------|
| **托管** | GitHub Pages | 免费静态托管，自动从 `main` 分支部署 |
| **前端** | 原生 HTML5 + CSS3 + ES6+ | 零依赖，单文件 `index.html` |
| **数据层** | `localStorage` | 离线缓存，即时读写 |
| **云端** | GitHub REST API v3 | 文件读写，base64 编码 |
| **版本控制** | Git + GitHub | 代码管理 + 数据版本历史 |
| **认证** | Fine-grained PAT + SHA-256 密码哈希 | 管理员鉴权 + API 访问 |
| **公共读取** | raw.githubusercontent.com CDN | 访客只读访问，无需 Token |

### 为什么选这些技术？

- **GitHub Pages** — 免费、国内可访问、自动 HTTPS、和仓库绑定
- **纯前端** — 无需服务器、无需构建、直接浏览器打开就能用
- **localStorage** — 同源策略保护、5-10MB 容量、无须后端
- **GitHub API** — 个人 Token 免费、5000 次/小时配额、数据天然有 git 历史

## 三、GitHub 配置详解

### 3.1 GitHub Pages 启用

```bash
# 1. 创建仓库（Public）
# 2. 推送代码到 main 分支
git remote add origin https://github.com/<user>/<repo>.git
git branch -M main
git push -u origin main

# 3. 在 GitHub 仓库 Settings → Pages 中配置：
#    Source: Deploy from a branch
#    Branch: main, / (root)
#    Save → 自动分配 URL
```

**访问地址**：`https://<user>.github.io/<repo>/`

### 3.2 Fine-grained Personal Access Token

**为什么用 Fine-grained 而不是 Classic？**
- Fine-grained 可以限制到**单个仓库**，更安全
- Fine-grained 权限粒度更细（Contents Read and write 即可）
- Fine-grained 没有过期时间（除非撤销）

**创建步骤：**
1. 打开 https://github.com/settings/tokens?type=beta
2. **Generate new token** → **Fine-grained token**
3. Token name: `命令管理器`
4. Repository access: **Only select repositories** → 选中你的仓库
5. Permissions: **Contents** → **Read and write**
6. 生成后 **立即复制**（只显示一次）

**Token 在代码中的使用：**

```javascript
// 存 localStorage（每个浏览器独立）
localStorage.setItem('github_token', token);

// 请求头格式
fetch(url, {
    headers: {
        Authorization: `Bearer ${token}`,    // Fine-grained 用 Bearer
        Accept: 'application/vnd.github.v3+json'
    }
});
```

### 3.3 GitHub API 文件读写

```javascript
// === 核心 API：仓库文件内容 ===
// GET: 读取文件（返回 base64 编码内容 + SHA）
GET /repos/{owner}/{repo}/contents/{path}

// PUT: 创建或更新文件
PUT /repos/{owner}/{repo}/contents/{path}
Body: {
    message: "commit message",
    content: base64(json),        // base64 编码的文件内容
    sha: "current_file_sha",      // 更新时必须传，创建时省略
    branch: "main"
}
```

**SHA 机制（重要）：**
- 每个文件在 git 中有一个唯一的 blob SHA
- 更新文件时必须提供当前 SHA，否则返回 **409 Conflict**
- 这个机制防止了并行写入的冲突（乐观锁）

**Base64 编码处理（注意 Unicode 兼容）：**

```javascript
// 写入：UTF-8 → base64（兼容中文）
function utf8ToBase64(str) {
    return btoa(unescape(encodeURIComponent(str)));
}

// 读取：base64 → UTF-8
function base64ToUtf8(str) {
    return decodeURIComponent(escape(atob(str)));
}
```

### 3.4 CORS 跨域注意事项

浏览器访问 `api.github.com` 属于跨域请求。关键点：

- `Authorization` 头会触发 CORS **预检请求**（OPTIONS）
- 请求头只应包含**必要**的头，不要加 `User-Agent`、`Cache-Control` 等
- 加 `?_=${Date.now()}` 时间戳参数防浏览器缓存过期 SHA

```javascript
// ✅ 正确姿势
const url = `https://api.github.com/.../data.json?_=${Date.now()}`;
fetch(url, {
    headers: {
        Authorization: `Bearer ${token}`,         // 必须
        Accept: 'application/vnd.github.v3+json' // 推荐
    }
});

// ❌ 避免的非标准头
// 'User-Agent': 'my-app'      → 禁止头，被浏览器忽略或阻塞
// 'Cache-Control': 'no-cache' → 触发额外预检，可能被拦截
```

## 四、前端架构模式

### 4.1 访客/管理员双模式

应用首次加载时自动进入**访客模式**，无需任何配置即可使用。通过密码登录切换为**管理员模式**。

```
页面加载
  → loadCommands()（localStorage）
  → isAdmin = localStorage.getItem('admin_logged_in') === 'true'
  → updateModeUI()

     ├─ 访客模式 ────────────────────────────
     │  fetchPublicData()                    ← CDN 极速拉取
     │    → raw.githubusercontent.com/.../data/data.json
     │    → 仅读取，无认证，跳过 CORS 预检
     │    → 本地无数据或数据过时时覆盖本地
     │
     └─ 管理员模式 ──────────────────────────
        updateSyncUI()
        syncPull()                           ← GitHub API 拉取
          → api.github.com/.../data/data.json
          → 需要 Token 认证，SHA 冲突检测
```

**切换方式：** CSS 类切换（不需要重新渲染）

```css
body:not(.admin-mode) .admin-only { display: none !important; }
body:not(.visitor-mode) .visitor-only { display: none !important; }
```

所有管理操作按钮添加 `class="admin-only"`，访客登录按钮和徽章添加 `class="visitor-only"`。

### 4.4 localStorage 读写模式

```javascript
// 读
function loadCommands() {
    const stored = localStorage.getItem('commandManagerData');
    if (stored) {
        commands = JSON.parse(stored);
        // 兼容旧数据...
    } else {
        commands = defaultCommands;
    }
}

// 写（每次修改后都调用，确保即时持久化）
function saveToStorage() {
    localStorage.setItem('commandManagerData', JSON.stringify(commands));
}
```

**关键设计：所有修改命令的函数都必须调用 `saveToStorage()`**

### 4.6 搜索模式（多词 + 模糊）

```javascript
function matchesSearch(cmd, terms) {
    // 每个词都要匹配（AND 逻辑）
    return terms.every(term => {
        // 优先精确匹配
        if (cmdExact.includes(term) || descExact.includes(term)) return true;
        // 降级模糊匹配（subsequence match）
        return fuzzyMatch(cmdText, term);
    });
}

function fuzzyMatch(text, pattern) {
    let pi = 0;  // pattern index
    for (let i = 0; i < text.length && pi < pattern.length; i++) {
        if (text[i] === pattern[pi]) pi++;
    }
    return pi === pattern.length;  // 所有字符按序找到
}
```

## 五、云同步核心逻辑

### 5.1 同步流程

```
页面加载
  ├─→ loadCommands()           ← 立即从 localStorage 加载
  ├─→ renderCommands()          ← 立即渲染
  ├─→ isAdmin? ─── 是 → syncPull()  ← GitHub API（需 Token）
  └─→ 否 → fetchPublicData()   ← raw CDN（无需认证）

管理员操作（增/删/改）
  → saveToStorage()             ← 先存本地（即时响应）
  → syncPush() [后台]           ← 再推云端（不阻塞 UI）

手动同步（仅管理员）
  → manualSync()
    → syncPull()                ← 先拉取云端最新
    → syncPush()                ← 再推送本地修改
```

### 5.2 syncPull — 拉取云端数据

```javascript
async function syncPull() {
    // 1. 从 GitHub 读取文件
    const result = await fetchGitHubFile(token);
    
    // 2. 只覆盖本地当云端有新数据
    if (result && result.content.commands.length > 0) {
        const ghTime = new Date(result.content.updatedAt);
        const localTime = localStorage.getItem('lastSyncAt');
        
        if (!localTime || ghTime > new Date(localTime)) {
            // 云端比本地新 → 替换本地
            commands = result.content.commands;
            saveToStorage();
            filterCommands();  // 重新渲染
        }
    }
}
```

### 5.3 syncPush — 推送本地数据

```javascript
async function syncPush() {
    // 1. 先获取当前文件的 SHA（防冲突）
    const current = await fetchGitHubFile(token);
    const sha = current?.sha;
    
    // 2. 尝试推送（409 冲突时自动重试）
    for (let retry = 0; retry < 3; retry++) {
        try {
            await pushGitHubFile(commands, token, sha);
            break;
        } catch (e) {
            if (retry < 2 && e.message.includes('409')) continue;
            throw e;
        }
    }
    
    // 3. 记录同步时间
    localStorage.setItem('lastSyncAt', new Date().toISOString());
}
```

### 5.4 并发控制

```javascript
let _syncing = false;  // 全局锁

async function syncPush() {
    if (_syncing) return;  // 已有同步进行中
    _syncing = true;
    try {
        // ... 同步逻辑
    } finally {
        _syncing = false;
    }
}
```

### 5.6 CDN 公共读取（访客模式）

```javascript
async function fetchPublicData() {
    const url = 'https://raw.githubusercontent.com/ironmole666/linux-command/main/data/data.json'
              + '?_=' + Date.now();
    const res = await fetch(url);
    if (!res.ok) return null;
    const data = await res.json();
    if (data.commands && data.commands.length > 0) {
        if (data.commands.length !== commands.length) {
            commands = data.commands.map(c => ({
                ...c, favorite: c.favorite || false, copyCount: c.copyCount || 0
            }));
        }
    }
}
```

`raw.githubusercontent.com` 特性：
- ✅ 无需 Token，无需认证
- ✅ 全球 CDN 加速，延迟低
- ✅ 没有 CORS 预检（简单 GET 请求，无自定义头）
- ✅ 动态时戳参数 `?_=` 防浏览器缓存
- ❌ 只读，不能写入

| 特性 | `api.github.com` | `raw.githubusercontent.com` |
|------|-----------------|---------------------------|
| 认证 | Bearer Token | 不需要 |
| 权限 | 读写 | 只读 |
| CORS | 预检（OPTIONS） | 无预检 |
| 速度 | 一般 | CDN 加速 |
| 配额限制 | 5000 次/小时 | 无限制 |

## 八、安全分析

> 本项目是纯前端单文件应用，**没有后端服务器**。理解其安全边界对正确使用至关重要。

### 8.1 认证机制的局限性

管理员登录**全部在浏览器 JS 中完成**，没有任何服务端验证：

```javascript
// handleLogin() — 完整的认证逻辑
async function handleLogin() {
    const username = document.getElementById('loginUsername').value.trim();
    const password = document.getElementById('loginPassword').value;
    const creds = getAdminCreds() || getCloudAdminCreds();
    const hash = await hashPassword(password);
    if (creds.username === username && creds.passwordHash === hash) {
        localStorage.setItem('admin_logged_in', 'true');  // ← 绕过点
        updateModeUI();  // ← 只切换 CSS 类
    }
}
```

**绕过方式：** 打开 DevTools → Application → Local Storage → 直接将 `admin_logged_in` 设为 `"true"` → 刷新页面即成为管理员。不需要知道密码，不需要破解哈希。

### 8.2 localStorage 中可见的数据

| Key | 内容 | 敏感程度 |
|-----|------|---------|
| `admin_creds` | `{ username, passwordHash }` | 哈希不可逆，但用户名明文可见 |
| `cloud_admin_creds` | 同上（从云端同步的副本） | 同上 |
| `admin_logged_in` | `"true"` 或不存在 | 直接决定权限 |
| `github_token` | `github_pat_xxxxxxxxxxxx` | **高敏感** — 持有者可读写仓库文件 |
| `commandManagerData` | 全部命令数据（JSON 数组） | 明文可见 |
| `categoryStructure` | 分类结构 | 低敏感 |
| `lastSyncAt` | 最后同步时间戳 | 低敏感 |

### 8.3 为什么可以接受这种设计

这个项目的定位是**个人命令备忘工具**，安全模型基于以下假设：

1. **威胁模型是防误操作，而非防攻击** — 防止路过的人随意修改/删除命令，而不是防止定向攻击
2. **数据本身不敏感** — 存的是 Linux 命令和开发工具命令，不是密码/密钥
3. **真正关键的防线是 GitHub Token** — 没有 Token，即使以管理员身份登录也无法修改云端数据；Token 被窃取才能造成持久影响
4. **GitHub 端的保护** — Token 是 Fine-grained PAT，限制到单个仓库的 Contents 读写权限；数据有 git 历史，可回滚

### 8.4 实际安全措施层级

```
第一层：CSS 类隔离
  body:not(.admin-mode) .admin-only { display: none !important; }
  → 防普通用户看到管理按钮
  → 绕过：F12 删除该 CSS 规则

第二层：JS 守卫
  N 快捷键中检查 isAdmin
  → 防键盘操作绕过
  → 绕过：直接在 Console 调 openModal()

第三层：密码验证（前端 SHA-256）
  → 防不知道密码的人直接操作
  → 绕过：DevTools 设 admin_logged_in = true

第四层：GitHub Token 验证（服务端）
  → 真正的写权限控制
  → 所有 API 请求带 Authorization: Bearer token
  → 绕过：窃取 Token
```

前两层是**防君子**，第三层是**轻微障碍**，只有第四层才是**真正的安全边界**。

### 8.5 密码哈希细节

```javascript
async function hashPassword(password) {
    const encoder = new TextEncoder();
    const data = encoder.encode(password);
    const hash = await crypto.subtle.digest('SHA-256', data);
    return Array.from(new Uint8Array(hash))
        .map(b => b.toString(16).padStart(2, '0')).join('');
}
```

- 使用 Web Crypto API `crypto.subtle.digest('SHA-256')`，浏览器原生实现
- 只在 HTTPS 下可用（GitHub Pages 自动满足）
- 输出 64 位十六进制字符串
- 没有加盐（salt）— 对于单用户场景可接受，但理论上彩虹表可还原常见密码
- 明文密码从不持久化到任何存储介质

### 8.6 Token 安全最佳实践

1. 使用 **Fine-grained PAT** 而非 Classic Token，限制到单个仓库
2. 权限只需 **Contents: Read and write**，不要多给
3. Token 只存在 `localStorage`，不硬编码，不写日志，不分享
4. 定期轮换 Token（GitHub 不设过期，但建议主动更换）
5. 如果怀疑 Token 泄露，立即在 GitHub Settings 中撤销并重新生成
6. Token 的写入操作会创建 commit，可在仓库 commit history 中追溯到操作时间

### 8.7 未来可做的安全增强

| 方案 | 说明 | 代价 |
|------|------|------|
| 后端认证代理 | Node.js/Cloudflare Worker 中转 API 请求，服务端校验密码 | 需要服务器，失去纯前端优势 |
| OAuth 登录 | GitHub OAuth 代替密码，无 Token 存储 | 需要后端，复杂度高 |
| 每次输入 Token | 不存 localStorage，每次打开页面手动输入 | 体验差，但最安全 |
| IndexedDB 加密 | 使用 Web Crypto API 加密 localStorage 数据 | 仍可被 DevTools 读取密钥 |
| Content Security Policy | 限制 XSS 攻击面 | 不能解决 localStorage 直接读取的问题 |

## 九、已知限制

| 限制 | 说明 | 解决方案/备注 |
|------|------|--------------|
| 单文件代码膨胀 | 全在一个 HTML 里，超过 1900 行 | 拆分成多个文件 + 构建工具 |
| 无冲突合并 | 后推送覆盖先推送 | 单用户场景下很少发生 |
| 无离线队列 | 离线时修改无法同步 | 加 IndexedDB + 同步队列 |
| 无实时推送 | 需要点同步或刷新页面 | 加 WebSocket / SSE |
| 前端认证绕过 | 任何人在 DevTools 设 `admin_logged_in=true` 即可登录 | 详细分析见 §8 安全分析；设计定位是防误操作而非防攻击 |
| Token 明文存储 | GitHub Token 明文在 localStorage 中 | 需手动保护电脑访问权限；定期轮换 Token |
| 密码无加盐 | SHA-256 哈希无 salt，彩虹表可还原常见密码 | 单用户场景下可接受；建议使用强密码 |

## 十、未来扩展思路

- **WebSocket 实时同步** — 一个设备修改后其他设备自动更新
- **PWA 离线支持** — Service Worker 缓存 + 离线编辑队列
- **多用户协作** — 后端 API + 数据库（Firebase / Supabase）
- **搜索语法增强** — 支持 `tag:xxx`、`cat:xxx` 等筛选语法
- **命令执行** — 集成 xterm.js 在浏览器中直接运行命令
- **AI 推荐** — 根据使用频率和标签推荐相似命令
